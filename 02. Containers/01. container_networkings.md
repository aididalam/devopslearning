# Container Networking From Scratch — Notes

## Most Important Sentence

> Docker bridge networking is not magic; it is built from Linux network namespaces, veth pairs, a bridge, routes, and iptables NAT.

---

# 1. Core Mental Model

```text
network namespace = isolated network room
veth pair         = virtual cable
bridge            = virtual switch
br0               = gateway for the container subnet
route             = packet direction map
ip_forward        = host acts as a router
MASQUERADE        = rewrite source IP for outbound traffic
DNAT              = rewrite destination IP for published ports
```

---

# 2. Final Network Topology

```text
Host / root namespace
│
├── wlan0 / eth0 → outside world
│
├── br0 = 172.18.0.1/16
│    ├── veth0 ─── ceth0 = 172.18.0.10/16 (netns0)
│    └── veth1 ─── ceth1 = 172.18.0.20/16 (netns1)
```

Packet flow:

```text
netns0 → ceth0 → veth0 ┐
                       ├── br0 → host → wlan0/eth0 → internet
netns1 → ceth1 → veth1 ┘
```

---

# 3. Network Namespace

Create isolated network environments.

```bash
sudo ip netns add netns0
sudo ip netns add netns1
```

Enter a namespace:

```bash
sudo nsenter --net=/run/netns/netns0 bash
```

A new namespace initially contains:

```text
- loopback interface only
- no routes
- separate network stack
```

---

# 4. veth Pair

A virtual Ethernet cable.

Create:

```bash
sudo ip link add veth0 type veth peer name ceth0
```

Meaning:

```text
veth0 <──── virtual cable ────> ceth0
```

Move one side into the namespace:

```bash
sudo ip link set ceth0 netns netns0
```

Now:

```text
Host side:      veth0
Container side: ceth0
```

Packets entering one side come out from the other side.

---

# 5. Assign IP Addresses

Host side:

```bash
sudo ip link set veth0 up
sudo ip addr add 172.18.0.11/16 dev veth0
```

Namespace side:

```bash
sudo nsenter --net=/run/netns/netns0 ip link set lo up
sudo nsenter --net=/run/netns/netns0 ip link set ceth0 up
sudo nsenter --net=/run/netns/netns0 ip addr add 172.18.0.10/16 dev ceth0
```

---

# 6. Automatic Routes

When assigning:

```bash
ip addr add 172.18.0.10/16 dev ceth0
```

Linux automatically creates:

```text
172.18.0.0/16 dev ceth0
```

Meaning:

```text
All 172.18.x.x traffic goes through ceth0
```

But there is still no default route.

---

# 7. Why Direct veth Connections Do Not Scale

If every host-side veth gets an IP:

```text
veth0 = 172.18.0.11/16
veth1 = 172.18.0.21/16
```

The host gets multiple identical routes:

```text
172.18.0.0/16 dev veth0
172.18.0.0/16 dev veth1
```

This creates routing ambiguity.

Solution:

```text
Use a Linux bridge
```

---

# 8. Linux Bridge

A bridge acts like a virtual switch.

Create:

```bash
sudo ip link add br0 type bridge
sudo ip link set br0 up
```

Attach interfaces:

```bash
sudo ip link set veth0 master br0
sudo ip link set veth1 master br0
```

Topology:

```text
          br0
        /     \
     veth0   veth1
       |       |
     ceth0   ceth1
```

Important:

```text
veth0/veth1 should NOT have IP addresses
```

IPs belong on:

```text
ceth0
ceth1
br0
```

---

# 9. br0 as Gateway

Assign IP to the bridge:

```bash
sudo ip addr add 172.18.0.1/16 dev br0
```

Purpose:

```text
Host becomes reachable from the container network
```

This creates:

```text
172.18.0.0/16 dev br0
```

---

# 10. Container Default Route

Containers only know local subnet routes.

Add default gateway:

```bash
sudo nsenter --net=/run/netns/netns0 \
  ip route add default via 172.18.0.1

sudo nsenter --net=/run/netns/netns1 \
  ip route add default via 172.18.0.1
```

Meaning:

```text
Unknown destinations → send to br0 gateway
```

---

# 11. Enable IP Forwarding

Allow the host to act as a router:

```bash
sudo sysctl -w net.ipv4.ip_forward=1
```

Without this:

```text
Host will not forward packets
```

---

# 12. MASQUERADE (Outbound NAT)

Containers use private IPs:

```text
172.18.x.x
```

The internet cannot route back to them.

Fix:

```bash
sudo iptables -t nat -A POSTROUTING \
  -s 172.18.0.0/16 ! -o br0 \
  -j MASQUERADE
```

Meaning:

```text
Rewrite container source IP
to the host's real outgoing IP
```

Before:

```text
SRC=172.18.0.10
DST=8.8.8.8
```

After:

```text
SRC=host-public-IP
DST=8.8.8.8
```

---

# 13. DNAT / Port Publishing

Expose a container port through the host.

Container app:

```text
172.18.0.10:5000
```

Host IP:

```text
192.168.110.61
```

Forward traffic:

```bash
sudo iptables -t nat -A PREROUTING \
  -d 192.168.110.61 \
  -p tcp --dport 5000 \
  -j DNAT \
  --to-destination 172.18.0.10:5000
```

Meaning:

```text
hostIP:5000 → containerIP:5000
```

---

# 14. OUTPUT DNAT

PREROUTING handles external packets.

But packets generated locally by the host skip PREROUTING.

So host-local traffic needs:

```bash
sudo iptables -t nat -A OUTPUT \
  -d 192.168.110.61 \
  -p tcp --dport 5000 \
  -j DNAT \
  --to-destination 172.18.0.10:5000
```

---

# 15. Important Packet Paths

External client:

```text
client
  ↓
PREROUTING
  ↓
DNAT
  ↓
container
```

Host-local client:

```text
host process
  ↓
OUTPUT
  ↓
DNAT
  ↓
container
```

Container to internet:

```text
container
  ↓
default route
  ↓
br0
  ↓
host forwarding
  ↓
POSTROUTING MASQUERADE
  ↓
internet
```

---

# 16. Important iptables Concepts

```text
-t nat
= use NAT table

-j
= action/target

DNAT
= change destination IP

SNAT
= change source IP

MASQUERADE
= dynamic SNAT
```

# 17. Common Debugging Commands

Interfaces:

```bash
ip link
```

Bridge members:

```bash
ip link show master br0
bridge link
```

Routes:

```bash
ip route
```

Namespaces:

```bash
ip netns list
```

iptables rules:

```bash
sudo iptables -t nat -L -n -v
```

Realtime interface events:

```bash
ip monitor link
```

Kernel networking logs:

```bash
sudo journalctl -kf
```

# 18. Cleanup Commands

```bash
sudo ip netns del netns0 2>/dev/null
sudo ip netns del netns1 2>/dev/null

sudo ip link del veth0 2>/dev/null
sudo ip link del veth1 2>/dev/null

sudo iptables -t nat -F
```

# 19. Docker Mapping

Docker bridge mode internally does roughly this:

```text
create network namespace
create veth pair
attach host side to bridge
assign container IP
add default route
enable NAT
publish ports with DNAT
```

Equivalent idea:

```text
Docker bridge networking =
netns + veth + bridge + route + NAT
```

# 20. Final Cheat Sheet

```text
netns       = isolated network room
veth pair   = virtual cable
bridge      = virtual switch
br0         = gateway
default route = send unknown traffic to gateway
ip_forward  = host becomes router
MASQUERADE  = outbound NAT
DNAT        = inbound port forwarding
```
