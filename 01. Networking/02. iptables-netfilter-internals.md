# iptables / Netfilter Revision Notes

## From Packet Walk to Stateful Firewall and NAT

---

# 1. Core Mental Model

Do not treat **iptables**, **netfilter**, **tables**, **chains**, and **rules** as the same thing.

```text
netfilter = Linux kernel packet-processing framework
iptables  = userspace tool used to configure netfilter rules
table     = category of packet-processing logic
chain     = ordered list of rules
rule      = match condition + target/action
target    = what to do when a rule matches
conntrack = kernel flow/connection tracking system
```

Most important sentence:

> iptables is the control tool; netfilter in the Linux kernel does the actual packet processing.

Simple mental model:

```text
Packet enters Linux
    ↓
netfilter hooks are triggered
    ↓
iptables rules are checked
    ↓
matching rule jumps to target
    ↓
packet is accepted, dropped, modified, logged, or forwarded
```

---

# 2. Why Packet Flow Comes First

iptables becomes easier when you stop memorizing commands and start walking like a packet.

There are three main packet paths:

```text
1. Packet comes from outside and goes to a local application.
2. Packet is created by a local application and goes outside.
3. Packet comes in one interface and leaves through another interface.
```

These three paths map to:

```text
INPUT   = traffic to this machine
OUTPUT  = traffic from this machine
FORWARD = traffic through this machine
```

Key point:

```text
Before writing iptables rules, first ask:
“Is this packet going to the host, from the host, or through the host?”
```

---

# 3. Netfilter Hooks

Netfilter hooks are packet-processing points inside the Linux kernel network stack.

Main hooks:

```text
PREROUTING
INPUT
FORWARD
OUTPUT
POSTROUTING
```

Simple meaning:

```text
PREROUTING   = packet just entered, before routing decision
INPUT        = packet is going to local process
FORWARD      = packet is being routed through the machine
OUTPUT       = packet was created by local process
POSTROUTING  = packet is about to leave
```

Key point:

```text
Chains are attached to packet path stages.
```

---

# 4. Incoming Packet to Local Application

Scenario:

```text
Laptop → Pi SSH
Client → Server HTTPS
Client → Pi netcat listener
```

Packet path:

```text
Network interface
    ↓
PREROUTING
    ↓
Routing decision
    ↓
INPUT
    ↓
Local application
```

Example:

```text
Client connects to server port 22.
The packet is for sshd running on the server.
So the packet goes through INPUT.
```

Key point:

```text
Traffic destined to local services hits INPUT.
```

---

# 5. Locally Generated Packet

Scenario:

```text
Pi → google.com
Server → apt update
Host → curl api.example.com
```

Packet path:

```text
Local application
    ↓
OUTPUT
    ↓
Routing decision
    ↓
POSTROUTING
    ↓
Network interface
```

Example:

```text
The server runs curl google.com.
The packet is created locally.
So it starts from OUTPUT.
```

Key point:

```text
Locally generated traffic hits OUTPUT, not PREROUTING.
```

---

# 6. Forwarded Packet

Scenario:

```text
Container namespace → Linux host → internet
LAN client → router → internet
VPN client → VPN server → private network
```

Packet path:

```text
Incoming interface
    ↓
PREROUTING
    ↓
Routing decision
    ↓
FORWARD
    ↓
POSTROUTING
    ↓
Outgoing interface
```

Example:

```text
A container sends traffic through the host to the internet.
The packet is not for the host itself.
The host is only routing it.
So the packet goes through FORWARD.
```

Key point:

```text
Router/gateway traffic hits FORWARD.
```

---

# 7. INPUT, OUTPUT, FORWARD

## INPUT

```text
Packet destination = this machine
```

Examples:

```text
SSH to server
HTTP request to nginx on server
Ping to server IP
Netcat connection to local listener
```

Useful phrase:

> INPUT protects services running on the box.

---

## OUTPUT

```text
Packet source = this machine
```

Examples:

```text
curl from server
apt update
DNS query from server
HTTP request from local process
```

Useful phrase:

> OUTPUT controls traffic generated by the box.

---

## FORWARD

```text
Packet is passing through this machine
```

Examples:

```text
Container → host → internet
LAN client → router → internet
VPN client → private network
```

Useful phrase:

> FORWARD controls routed traffic through the box.

---

# 8. Routing Decision

The routing decision decides:

```text
Is this packet for this machine?
Should this packet be forwarded?
Which interface should it leave from?
Which next hop should be used?
```

After `PREROUTING`:

```text
If destination is local:
  → INPUT

If destination is not local and forwarding is enabled:
  → FORWARD
```

For locally generated traffic:

```text
OUTPUT
  ↓
Routing decision
  ↓
POSTROUTING
```

Key point:

```text
Routing decision chooses INPUT vs FORWARD.
```

---

# 9. Tables

iptables has multiple tables.

Each table has a different purpose.

```text
filter   = allow / drop / reject
nat      = source or destination IP/port rewrite
mangle   = packet marking or modification
raw      = conntrack bypass / NOTRACK
security = MAC / SELinux-style security decision
```

Simple memory:

```text
raw      = track korbo naki korbo na
mangle   = packet mark/modify korbo
nat      = IP/port identity bodlabo
filter   = allow/drop/reject korbo
security = filter er pore extra security decision
```

Key point:

```text
Table decides the type of work.
Chain decides where in the packet path that work happens.
```

---

# 10. filter Table

The `filter` table is used for normal firewall decisions.

Purpose:

```text
Allow packet
Drop packet
Reject packet
```

Built-in chains:

```text
INPUT
FORWARD
OUTPUT
```

Examples:

```bash
iptables -A INPUT -p tcp --dport 22 -j ACCEPT
iptables -A INPUT -s 203.0.113.10 -j DROP
iptables -A FORWARD -i lan0 -o wan0 -j ACCEPT
iptables -A OUTPUT -p tcp --dport 443 -j ACCEPT
```

Mental model:

```text
filter INPUT   = should this packet reach local application?
filter OUTPUT  = should this local application send this packet?
filter FORWARD = should this packet pass through this machine?
```

Key point:

```text
filter table answers:
“Should this packet be allowed?”
```

---

# 11. nat Table

The `nat` table is used to rewrite packet identity.

Packet identity can mean:

```text
source IP
source port
destination IP
destination port
```

Common targets:

```text
DNAT
SNAT
MASQUERADE
REDIRECT
```

Important chains:

```text
PREROUTING   = destination rewrite before routing decision
OUTPUT       = NAT for locally generated packets
POSTROUTING  = source rewrite before packet leaves
```

Key point:

```text
nat table answers:
“Should this packet look like it came from somewhere else or is going somewhere else?”
```

---

# 12. DNAT

DNAT means **Destination NAT**.

It changes:

```text
destination IP
destination port
```

Usually used in:

```text
PREROUTING
```

Example:

```bash
iptables -t nat -A PREROUTING -p tcp --dport 8080 \
  -j DNAT --to-destination 10.10.10.5:80
```

Before DNAT:

```text
Client → router:8080
```

After DNAT:

```text
Client → 10.10.10.5:80
```

Use case:

```text
Port forwarding to another host
```

Key point:

```text
DNAT usually happens before routing because the kernel must know the final destination before choosing a route.
```

---

# 13. SNAT

SNAT means **Source NAT**.

It changes:

```text
source IP
source port if needed
```

Usually used in:

```text
POSTROUTING
```

Example:

```bash
iptables -t nat -A POSTROUTING -s 10.10.10.0/24 -o eth0 \
  -j SNAT --to-source 203.0.113.10
```

Before SNAT:

```text
10.10.10.5 → internet
```

After SNAT:

```text
203.0.113.10 → internet
```

Use case:

```text
Private network accessing internet through static public IP
```

Key point:

```text
SNAT usually happens after routing because the kernel first decides the outgoing interface.
```

---

# 14. MASQUERADE

MASQUERADE is dynamic source NAT.

Example:

```bash
iptables -t nat -A POSTROUTING -s 10.10.10.0/24 -o wlan0 \
  -j MASQUERADE
```

Meaning:

```text
Rewrite source IP to the current IP of outgoing interface.
```

Used when:

```text
WAN IP is dynamic
DHCP interface
Wi-Fi uplink
home router
container internet access
```

Difference:

```text
SNAT        = use a specific source IP
MASQUERADE  = use current IP of outgoing interface
```

Key point:

```text
MASQUERADE is dynamic SNAT.
```

---

# 15. REDIRECT

REDIRECT is local destination NAT.

Example:

```bash
iptables -t nat -A PREROUTING -p tcp --dport 8080 \
  -j REDIRECT --to-ports 5343
```

Before REDIRECT:

```text
Client → server:8080
```

After REDIRECT:

```text
Client → server:5343
```

The destination remains the local machine, but the port changes.

Use cases:

```text
transparent proxy
local port redirect
service interception
testing with netcat
```

Key point:

```text
REDIRECT is DNAT to the local machine.
```

---

# 16. mangle Table

The `mangle` table is used for special packet alteration.

Used for:

```text
MARK
CONNMARK
TTL changes
TOS / DSCP changes
QoS
policy routing
advanced packet metadata changes
```

Example:

```bash
iptables -t mangle -A PREROUTING -s 10.10.10.5 \
  -j MARK --set-mark 10
```

Then Linux routing can use the mark:

```bash
ip rule add fwmark 10 table 100
```

Mental model:

```text
mangle does not usually answer allow/drop.
mangle marks or modifies the packet.
```

Key point:

```text
mangle table answers:
“Should I mark or modify this packet metadata?”
```

---

# 17. raw Table

The `raw` table is used before conntrack.

Main purpose:

```text
Decide whether a packet should be tracked or not.
```

Common target:

```text
NOTRACK
```

Example:

```bash
iptables -t raw -A PREROUTING -p udp --dport 123 -j NOTRACK
```

Meaning:

```text
Do not track this packet in conntrack.
```

Use carefully because conntrack is needed for:

```text
stateful firewalling
NAT mappings
ESTABLISHED / RELATED matching
```

Key point:

```text
raw table answers:
“Should conntrack track this packet?”
```

---

# 18. security Table

The `security` table is used for Mandatory Access Control networking rules.

Used with:

```text
SELinux
SECMARK
CONNSECMARK
MAC security policies
```

It is evaluated after the `filter` table.

Mental model:

```text
filter   = normal firewall decision
security = extra MAC/SELinux-style security decision
```

Key point:

```text
A packet accepted by filter can still be checked by security.
```

---

# 19. Table Summary

| Table    | Main Purpose                         | Simple Memory                 |
| -------- | ------------------------------------ | ----------------------------- |
| raw      | conntrack bypass / NOTRACK           | track korbo naki korbo na     |
| mangle   | packet mark or metadata modification | packet mark/modify korbo      |
| nat      | IP/port rewrite                      | packet identity bodlabo       |
| filter   | allow/drop/reject                    | packet allow korbo naki drop  |
| security | MAC/SELinux security                 | filter er pore extra security |

Important:

```text
Same chain name can exist in multiple tables.
```

Example:

```text
filter INPUT
mangle INPUT
security INPUT
```

These are not the same chain.

---

# 20. Same Chain Name in Different Tables

Example:

```text
mangle INPUT
filter INPUT
security INPUT
```

They are different chains attached around the local-input path.

Meaning:

```text
mangle INPUT   = mark/modify packet before local delivery
filter INPUT   = allow/drop/reject before local delivery
security INPUT = extra MAC decision after filter
```

Important:

```text
ACCEPT in mangle INPUT does not bypass filter INPUT.
```

Key point:

```text
Table + chain together define the real location and purpose.
```

Example:

```text
nat PREROUTING    = destination rewrite before routing
filter INPUT      = allow/drop before local process
mangle POSTROUTING = modify/mark before leaving
```

---

# 21. Rules

A rule is:

```text
match condition + target/action
```

Example:

```bash
iptables -A INPUT -s 192.168.1.10 -p tcp --dport 22 -j ACCEPT
```

Breakdown:

```text
-A INPUT              = append rule to INPUT chain
-s 192.168.1.10       = source IP match
-p tcp                = protocol match
--dport 22            = destination port match
-j ACCEPT             = target/action
```

Meaning:

```text
If packet comes from 192.168.1.10,
uses TCP,
and destination port is 22,
then accept it.
```

Key point:

```text
iptables rules are if-this-then-that logic for packets.
```

---

# 22. Rule Order

Rules are checked:

```text
top → bottom
```

Bad example:

```bash
iptables -A INPUT -j DROP
iptables -A INPUT -p tcp --dport 22 -j ACCEPT
```

Meaning:

```text
Drop everything first.
SSH allow rule is never reached.
```

Good example:

```bash
iptables -A INPUT -p tcp --dport 22 -j ACCEPT
iptables -A INPUT -j DROP
```

Meaning:

```text
Allow SSH first.
Drop everything else later.
```

Key point:

```text
Rule order is part of firewall logic.
```

# 23. Common Match Conditions

## Source IP

```bash
-s 192.168.1.10
```

## Destination IP

```bash
-d 10.0.0.5
```

## Protocol

```bash
-p tcp
-p udp
-p icmp
```

## Destination port

```bash
--dport 22
```

## Source port

```bash
--sport 12345
```

## Input interface

```bash
-i eth0
```

## Output interface

```bash
-o wlan0
```

## Conntrack state

```bash
-m conntrack --ctstate ESTABLISHED,RELATED
```

## Multiple ports

```bash
-m multiport --dports 80,443
```

Important:

```text
-i = interface packet came in from
-o = interface packet will go out through
```

# 24. Targets

A target says what happens when a rule matches.

Common targets:

```text
ACCEPT
DROP
REJECT
LOG
RETURN
DNAT
SNAT
MASQUERADE
REDIRECT
MARK
NOTRACK
```

## ACCEPT

```text
Allow packet at this point.
```

## DROP

```text
Silently discard packet.
```

## REJECT

```text
Reject packet and notify sender.
```

## LOG

```text
Log packet and continue to next rule.
```

## RETURN

```text
Return from current chain to caller chain.
```

## DNAT

```text
Change destination IP/port.
```

## SNAT

```text
Change source IP/port.
```

## MASQUERADE

```text
Dynamic source NAT.
```

## REDIRECT

```text
Redirect to local machine/port.
```

## MARK

```text
Mark packet for routing/QoS/advanced handling.
```

## NOTRACK

```text
Skip conntrack.
```

---

# 25. Terminating and Non-Terminating Targets

Some targets stop chain processing.

Terminating targets:

```text
ACCEPT
DROP
REJECT
RETURN
```

Non-terminating target:

```text
LOG
```

Example:

```bash
iptables -A INPUT -j LOG --log-prefix "INPUT "
iptables -A INPUT -j DROP
```

Meaning:

```text
First rule logs the packet.
Then packet continues to next rule.
Second rule drops it.
```

Important:

```text
LOG does not accept or drop.
LOG only logs and continues.
```

---

# 26. ACCEPT Is Not Always the Final End

Important subtle point:

```text
ACCEPT means packet is accepted in the current chain/hook.
```

But the packet may still pass through later hooks.

Example:

```text
Packet accepted in mangle PREROUTING
can still be dropped later in filter INPUT.
```

So always ask:

```text
Which table?
Which chain?
Which hook?
What stages are still ahead?
```

Key point:

```text
ACCEPT is local to the current packet-processing stage.
```

---

# 27. DROP vs REJECT

## DROP

```text
Silently discard packet.
Sender usually waits until timeout.
```

Example:

```bash
iptables -A INPUT -p tcp --dport 23 -j DROP
```

Use case:

```text
Internet-facing unwanted traffic
```

---

## REJECT

```text
Reject packet and notify sender.
Sender gets fast failure.
```

Example:

```bash
iptables -A INPUT -p tcp --dport 23 -j REJECT
```

Use case:

```text
Internal network debugging
clear failure behavior
```

Comparison:

```text
DROP   = no reply
REJECT = explicit no
```

---

# 28. Chain Policy

Built-in chains can have default policies.

Example:

```bash
iptables -P INPUT DROP
iptables -P FORWARD DROP
iptables -P OUTPUT ACCEPT
```

Meaning:

```text
If no INPUT rule matches   → DROP
If no FORWARD rule matches → DROP
If no OUTPUT rule matches  → ACCEPT
```

Important:

```text
Only built-in chains have policies.
User-defined chains do not have policies.
```

Key point:

```text
Policy is the default action at the end of a built-in chain.
```

---

# 29. Default Deny Firewall

Default deny means:

```text
Allow only required traffic.
Drop everything else.
```

Common baseline:

```bash
iptables -A INPUT -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT
iptables -A INPUT -i lo -j ACCEPT
iptables -A INPUT -m conntrack --ctstate INVALID -j DROP
iptables -A INPUT -p tcp --dport 22 -j ACCEPT
iptables -P INPUT DROP
```

Meaning:

```text
Allow replies to existing connections.
Allow loopback.
Drop invalid packets.
Allow SSH.
Drop all other incoming traffic.
```

Important warning:

```text
Never set INPUT DROP over SSH before allowing SSH.
```

---

# 30. User-Defined Chains

A user-defined chain is like a named function.

Example:

```bash
iptables -N LOG_DROP

iptables -A INPUT -j LOG_DROP

iptables -A LOG_DROP -m limit --limit 5/min --limit-burst 10 \
  -j LOG --log-prefix "DROP "

iptables -A LOG_DROP -j DROP
```

Flow:

```text
INPUT
  ↓ jump
LOG_DROP
  ↓ log
  ↓ drop
```

Important:

```text
User-defined chain is not a new kernel packet stage.
It is only a reusable ordered rule list.
```

Key point:

```text
Built-in chains are packet path stages.
User-defined chains are helper chains.
```

---

# 31. RETURN

`RETURN` means:

```text
Stop processing current chain.
Go back to the caller chain.
Continue from the next rule after the jump.
```

Example:

```bash
iptables -N CHECK_SSH

iptables -A INPUT -p tcp --dport 22 -j CHECK_SSH

iptables -A CHECK_SSH -s 192.168.1.10 -j ACCEPT
iptables -A CHECK_SSH -j RETURN

iptables -A INPUT -p tcp --dport 22 -j DROP
```

Flow:

```text
SSH packet enters INPUT
    ↓
jump to CHECK_SSH
    ↓
if source is trusted → ACCEPT
    ↓
otherwise RETURN to INPUT
    ↓
next INPUT rule drops SSH
```

Key point:

```text
RETURN is like returning from a function call.
```

---

# 32. Conntrack

Conntrack means **connection tracking**.

Without conntrack:

```text
Firewall sees individual packets only.
```

With conntrack:

```text
Firewall understands packet belongs to a flow.
```

Common conntrack states:

```text
NEW          = new connection attempt
ESTABLISHED  = packet belongs to existing connection
RELATED      = packet is related to an existing connection
INVALID      = packet cannot be associated with valid flow
UNTRACKED    = packet is intentionally not tracked
```

Key point:

```text
Conntrack turns packet filtering into stateful firewalling.
```

---

# 33. NEW

`NEW` means:

```text
This packet is trying to start a new connection.
```

Example:

```bash
iptables -A INPUT -p tcp --dport 22 \
  -m conntrack --ctstate NEW -j ACCEPT
```

Meaning:

```text
Allow new SSH connection attempts.
```

Often combined with:

```text
source IP restriction
specific port
rate limiting
logging
```

Example:

```bash
iptables -A INPUT -s 192.168.1.50 -p tcp --dport 22 \
  -m conntrack --ctstate NEW -j ACCEPT
```

---

# 34. ESTABLISHED and RELATED

Most important stateful firewall rule:

```bash
iptables -A INPUT -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT
```

Meaning:

```text
If this packet belongs to an already allowed connection,
allow it.
```

Example:

```text
Server sends request to google.com.
Google sends reply.
Reply enters INPUT.
Without ESTABLISHED rule, default DROP may block the reply.
```

`RELATED` means:

```text
Packet is not part of the same connection,
but is related to an existing connection.
```

Key point:

```text
ESTABLISHED,RELATED makes default deny firewall usable.
```

---

# 35. INVALID

`INVALID` means:

```text
Conntrack cannot associate packet with a valid flow.
```

Possible reasons:

```text
malformed packet
stale state
asymmetric routing
out-of-window TCP packet
conntrack table pressure
```

Common rule:

```bash
iptables -A INPUT -m conntrack --ctstate INVALID -j DROP
```

Production nuance:

```text
Dropping INVALID is common,
but check counters/logs before assuming attack.
```

Key point:

```text
INVALID packets are suspicious or not useful for normal stateful flow.
```

---

# 36. NAT and Conntrack

NAT depends on conntrack.

Example:

```text
client 10.10.10.2 → internet 1.1.1.1
```

Before NAT:

```text
SRC = 10.10.10.2
DST = 1.1.1.1
```

After MASQUERADE:

```text
SRC = router_wan_ip
DST = 1.1.1.1
```

Reply from internet:

```text
SRC = 1.1.1.1
DST = router_wan_ip
```

Conntrack remembers:

```text
This reply belongs to original 10.10.10.2 flow.
Rewrite destination back to 10.10.10.2.
```

Key point:

```text
NAT is flow-aware rewriting maintained through conntrack.
```

---

# 37. Why DNAT Happens Before Routing

DNAT changes destination.

The kernel must know the final destination before deciding where to send the packet.

Example:

```bash
iptables -t nat -A PREROUTING -p tcp --dport 8080 \
  -j DNAT --to-destination 10.10.10.5:80
```

Reason:

```text
Destination must be rewritten before routing decision.
```

Memory:

```text
DNAT / REDIRECT → PREROUTING
```

Key point:

```text
Change destination before route lookup.
```

---

# 38. Why SNAT Happens After Routing

SNAT changes source.

The kernel usually decides the outgoing interface first.

Example:

```bash
iptables -t nat -A POSTROUTING -o eth0 \
  -j MASQUERADE
```

Reason:

```text
Source can be rewritten based on outgoing interface.
```

Memory:

```text
SNAT / MASQUERADE → POSTROUTING
```

Key point:

```text
Change source after route lookup, before packet leaves.
```

---

# 39. Local Port Redirect

Goal:

```text
Incoming 8080 should go to local 5343.
```

Rule:

```bash
iptables -t nat -A PREROUTING -i wlan0 -p tcp --dport 8080 \
  -j REDIRECT --to-ports 5343
```

Packet flow:

```text
Client → server:8080
    ↓
nat PREROUTING
    ↓
REDIRECT 8080 → 5343
    ↓
routing decision: local machine
    ↓
INPUT
    ↓
local process listening on 5343
```

Test:

```bash
nc -lvnp 5343
nc -vz server 8080
```

Key point:

```text
Client thinks it connected to 8080.
Application receives traffic on 5343.
```

# 40. Linux as Router

To make Linux act as a router:

```text
1. Enable IP forwarding.
2. Allow traffic in FORWARD.
3. Add NAT if private clients need internet access.
```

Enable forwarding:

```bash
sudo sysctl -w net.ipv4.ip_forward=1
```

Forward rules:

```bash
iptables -A FORWARD -i lan0 -o wan0 \
  -m conntrack --ctstate NEW,ESTABLISHED,RELATED -j ACCEPT

iptables -A FORWARD -i wan0 -o lan0 \
  -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT
```

NAT:

```bash
iptables -t nat -A POSTROUTING -o wan0 -j MASQUERADE
```

Packet flow:

```text
LAN client
  ↓
lan0
  ↓
PREROUTING
  ↓
FORWARD
  ↓
POSTROUTING + MASQUERADE
  ↓
wan0
  ↓
internet
```

Key point:

```text
Router traffic needs FORWARD, not INPUT.
```

# 41. iptables and Containers

Container networking uses the same concepts.

Common pieces:

```text
network namespace
veth pair
Linux bridge
routing
FORWARD chain
NAT / MASQUERADE
conntrack
```

Typical container egress:

```text
container process
  ↓
container eth0
  ↓
veth pair
  ↓
host bridge
  ↓
host routing
  ↓
FORWARD
  ↓
POSTROUTING MASQUERADE
  ↓
internet
```

Important:

```text
Container traffic usually does not hit host INPUT.
It usually hits FORWARD when routed through host.
```

Key point:

```text
Understanding iptables helps understand Docker and Kubernetes networking.
```

# 43. LOG and LOG_DROP Pattern

`LOG` only logs.

It does not stop packet processing.

Bad assumption:

```text
LOG means packet is blocked.
```

Correct:

```text
LOG logs and continues.
```

Reusable logging drop chain:

```bash
iptables -N LOG_DROP

iptables -A LOG_DROP -m limit --limit 5/min --limit-burst 10 \
  -j LOG --log-prefix "DROP " --log-level 4

iptables -A LOG_DROP -j DROP

iptables -A INPUT -j LOG_DROP
```

Why rate limit?

```text
Without rate limit, scans or attacks can flood logs.
```

Key point:

```text
LOG + DROP is a common pattern.
```

# 44. SSH Safe Firewall Pattern

Safe order:

```bash
# Allow already established traffic
iptables -A INPUT -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT

# Allow loopback
iptables -A INPUT -i lo -j ACCEPT

# Drop invalid
iptables -A INPUT -m conntrack --ctstate INVALID -j DROP

# Allow SSH from trusted source
iptables -A INPUT -s <your_ip> -p tcp --dport 22 \
  -m conntrack --ctstate NEW -j ACCEPT

# Default deny
iptables -P INPUT DROP
```

Important:

```text
Do not set INPUT DROP first over SSH.
```

Good production habit:

```bash
iptables-save > /root/iptables.backup
```

Rollback idea:

```bash
echo "iptables-restore < /root/iptables.backup" | at now + 2 minutes
```

Key point:

```text
Firewall changes over SSH need rollback planning.
```

# 45. SSH Rate Limiting

Basic example:

```bash
iptables -A INPUT -p tcp --dport 22 -m conntrack --ctstate NEW \
  -m limit --limit 3/min --limit-burst 5 -j ACCEPT

iptables -A INPUT -p tcp --dport 22 -j DROP
```

Meaning:

```text
Allow limited new SSH connections.
Drop excessive new SSH attempts.
```

More source-aware idea:

```bash
iptables -A INPUT -p tcp --dport 22 -m conntrack --ctstate NEW \
  -m recent --set --name SSH

iptables -A INPUT -p tcp --dport 22 -m conntrack --ctstate NEW \
  -m recent --update --seconds 60 --hitcount 6 --name SSH -j DROP

iptables -A INPUT -p tcp --dport 22 -j ACCEPT
```

Security note:

```text
iptables rate limiting is helpful,
but SSH security also needs key-only auth,
disabled password login,
source restriction,
and monitoring.
```

Key point:

```text
Rate limiting controls connection attempts, not authentication quality.
```

# 47. Debugging Methodology

Do not randomly change firewall rules.

Debug by walking the packet.

## Step 1: Is the service listening?

```bash
ss -tulpen
```

Check:

```text
Is the application listening?
Which port?
Which address?
IPv4 or IPv6?
```

---

## Step 2: Is routing correct?

```bash
ip addr
ip route
ip rule
```

Check:

```text
Is destination local or forwarded?
Which interface will be used?
Which gateway?
```

---

## Step 3: Is forwarding enabled?

```bash
sysctl net.ipv4.ip_forward
```

Enable temporarily:

```bash
sudo sysctl -w net.ipv4.ip_forward=1
```

---

## Step 4: Inspect rules

```bash
iptables -S
iptables -t nat -S
iptables -t mangle -S
iptables -t raw -S
```

---

## Step 5: Inspect counters

```bash
iptables -L -n -v --line-numbers
iptables -t nat -L -n -v --line-numbers
```

Interpretation:

```text
counter increasing     = packet hits this rule
counter not increasing = packet is not reaching this rule/path
```

---

## Step 6: Capture packets

```bash
tcpdump -ni any host <ip>
tcpdump -ni eth0 port 80
tcpdump -ni wlan0 icmp
```

---

## Step 7: Check conntrack

```bash
conntrack -L
conntrack -L | grep <ip>
```

Look for:

```text
NEW
ESTABLISHED
NAT mappings
stale flows
```

---

## Step 8: Check logs

```bash
journalctl -k
dmesg
```

If using log prefix:

```bash
journalctl -k | grep "DROP"
```

Key point:

```text
tcpdump shows packet reality.
iptables counters show rule reality.
routing table shows kernel decision reality.
```

# 48. Debugging Questions

When traffic fails, ask:

```text
1. Is the packet local inbound, local outbound, or forwarded?
```

# 49. Common Mistakes

## Mistake 1: Confusing INPUT and FORWARD

Wrong:

```text
Container traffic not working, so check INPUT only.
```

Correct:

```text
Container/router traffic usually hits FORWARD.
```

---

## Mistake 2: Forgetting ESTABLISHED,RELATED

Wrong:

```bash
iptables -P INPUT DROP
```

without:

```bash
iptables -A INPUT -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT
```

Problem:

```text
Reply traffic may be dropped.
```

---

## Mistake 3: Dropping before allowing

Wrong:

```bash
iptables -A INPUT -j DROP
iptables -A INPUT -p tcp --dport 22 -j ACCEPT
```

Correct:

```bash
iptables -A INPUT -p tcp --dport 22 -j ACCEPT
iptables -A INPUT -j DROP
```

---

## Mistake 4: Thinking LOG drops packets

Wrong:

```text
LOG blocks traffic.
```

Correct:

```text
LOG only logs and continues.
```

---

## Mistake 5: Thinking ACCEPT in one table means final allow

Wrong:

```text
mangle ACCEPT means packet cannot be dropped later.
```

Correct:

```text
Packet can still be dropped later by filter/security.
```

---

## Mistake 6: NAT without forwarding

Problem:

```text
MASQUERADE exists,
but ip_forward=0
or FORWARD chain blocks traffic.
```

Correct:

```bash
sysctl -w net.ipv4.ip_forward=1

iptables -A FORWARD -i lan0 -o wan0 \
  -m conntrack --ctstate NEW,ESTABLISHED,RELATED -j ACCEPT

iptables -A FORWARD -i wan0 -o lan0 \
  -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT

iptables -t nat -A POSTROUTING -o wan0 -j MASQUERADE
```

---

## Mistake 7: Wrong loopback interface

Wrong:

```bash
iptables -A INPUT -i -lo -j ACCEPT
```

Correct:

```bash
iptables -A INPUT -i lo -j ACCEPT
```

# 52. Common Questions and Good Answers

## Q1. What is iptables?

iptables is a userspace tool used to configure packet filtering, NAT, and packet manipulation rules in the Linux kernel’s netfilter framework.

Better answer:

```text
iptables is the control plane.
netfilter is the kernel data plane.
Packets are processed in kernel space.
```

---

## Q2. Explain packet flow.

There are three main paths:

```text
Inbound local:
  PREROUTING → INPUT → local process

Local outbound:
  OUTPUT → POSTROUTING

Forwarded:
  PREROUTING → FORWARD → POSTROUTING
```

Routing decision after `PREROUTING` decides whether incoming packet goes to `INPUT` or `FORWARD`.

---

## Q3. INPUT vs FORWARD?

```text
INPUT   = packet is destined to local service
FORWARD = packet is routed through the machine
```

Example:

```text
SSH to server        → INPUT
Container to internet → FORWARD
```

---

## Q4. filter vs nat?

```text
filter = allow/drop/reject
nat    = source/destination IP/port rewrite
```

Example:

```text
Allow SSH        → filter INPUT
Port forwarding  → nat PREROUTING
Internet sharing → nat POSTROUTING MASQUERADE
```

---

## Q5. Why allow ESTABLISHED,RELATED?

Because reply packets for already allowed connections must be allowed.

Example:

```text
Server sends outbound request.
Reply comes back through INPUT.
Default DROP would block it unless ESTABLISHED is allowed.
```

---

## Q6. What is INVALID?

`INVALID` means conntrack cannot associate the packet with a valid flow.

Possible reasons:

```text
malformed packet
stale connection
asymmetric routing
out-of-window TCP packet
conntrack issue
```

Common action:

```text
Drop INVALID early,
but monitor counters/logs in production.
```

---

## Q7. What is MASQUERADE?

MASQUERADE is dynamic source NAT.

```text
It rewrites source IP to the current IP of outgoing interface.
```

Useful for:

```text
DHCP
Wi-Fi
home router
container internet access
```

---

## Q8. What is REDIRECT?

REDIRECT is local destination NAT.

Example:

```text
server:8080 → server:5343
```

Useful for:

```text
transparent proxy
local port redirect
testing
```

---

## Q9. Why DNAT in PREROUTING?

Because destination must be changed before routing decision.

```text
Kernel needs final destination before choosing route.
```

---

## Q10. Why SNAT in POSTROUTING?

Because source is usually rewritten after the route and outgoing interface are known.

```text
Kernel decides egress interface first.
Then source IP is rewritten.
```

---

## Q11. What is mangle used for?

```text
packet marking
TTL changes
DSCP/TOS
QoS
policy routing
advanced packet metadata changes
```

---

## Q12. What is raw used for?

```text
raw is evaluated before conntrack.
It is mainly used for NOTRACK.
```

---

## Q13. What is security table?

```text
security table is used for MAC/SELinux-style networking rules.
It is evaluated after filter.
```

---

## Q14. How do you debug iptables?

```text
1. Check service: ss -tulpen
2. Check route: ip route, ip addr
3. Check forwarding: sysctl net.ipv4.ip_forward
4. Check rules: iptables -S
5. Check counters: iptables -L -n -v
6. Check NAT: iptables -t nat -L -n -v
7. Capture packets: tcpdump
8. Check conntrack: conntrack -L
9. Check logs: journalctl -k
```

Best answer:

```text
I debug by walking the packet path.
```

# 54. Finally

> iptables is not just a firewall command tool. It is a way to program packet decisions inside the Linux kernel’s netfilter framework. The most important skill is walking the packet. If the packet is going to a local service, think `PREROUTING → INPUT`. If it is created locally, think `OUTPUT → POSTROUTING`. If it is routed through the machine, think `PREROUTING → FORWARD → POSTROUTING`. The `filter` table decides allow/drop/reject, the `nat` table rewrites source or destination identity, the `mangle` table marks or modifies packets, the `raw` table controls conntrack participation, and the `security` table applies extra MAC-style rules after filtering. Conntrack makes iptables stateful by remembering flows, which is why `ESTABLISHED,RELATED` is essential for default-deny firewalls and why NAT replies can be mapped back correctly. To debug iptables, do not guess rules; walk the packet, check routing, inspect counters, verify conntrack, and confirm reality with tcpdump.
